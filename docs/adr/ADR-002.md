# ADR-002: Dynamic Orchestration Architecture

## Context
Historically, browser extensions rely on hardcoded scripts for specific tasks (e.g., "dark mode", "hide shorts"). When transitioning to an AI-driven model (Manifest V3), executing raw JavaScript strings generated by an LLM via `eval()` or `<script>` injection is completely blocked by Content Security Policy (CSP) for security reasons. 

If a user asks for arbitrary functionality (e.g., "play a thunder sound and make the screen flash"), the system previously failed because it was locked into a static CSS `style-injection` format.

## Decision
To achieve true dynamic behavior without hardcoding features and without breaking MV3 CSP constraints, we have implemented the **Dynamic Action Interpreter** pattern.

Instead of the LLM generating raw JavaScript, it acts as a "compiler", generating a structured JSON array of secure DOM instructions. The Content Script acts as an "interpreter", safely executing these instructions.

### Schema Definition (Format 3)
```json
{
  "morphType": "dynamic-action",
  "actions": [
    { 
      "command": "createElement", 
      "tag": "audio", 
      "id": "thunder-audio", 
      "attributes": { "src": "https://example.com/sound.mp3", "autoplay": "true", "loop": "true" } 
    },
    { 
      "command": "appendChild", 
      "parentId": "body", 
      "targetId": "thunder-audio" 
    }
  ]
}
```

## Consequences
**Positive:**
- **Zero `eval()`**: Fully compliant with Manifest V3. No arbitrary string execution.
- **Sandboxed Execution**: Only explicit, whitelisted DOM actions can be executed (`createElement`, `setAttribute`, `appendChild`, `removeElement`).
- **Reversibility**: The `DynamicMorph` tracks all created elements and mutations, allowing for an instant, clean revert of any dynamically generated functionality.
- Vastly expands AI capabilities beyond static CSS (e.g., enabling sounds, media, and dynamic elements).

**Negative:**
- The DOM instruction set introduces a small execution overhead and requires strict JSON schema adherence from the LLM.
- Cannot run complex, stateful arbitrary JS logic (like nested loops or custom event listeners) natively inside the page, but is heavily optimized for declarative rendering and composition.

## Alternatives considered
- **Sandboxed Iframes (`sandbox` property)**: Allows `eval()`, but cannot easily mutate the main DOM context where the website layout lives. Rejected because it breaks the "morphing" aspect of altering the host page directly.
- **Hardcoded Capability Morphs (e.g., specific AudioMorph, VideoMorph classes)**: Rejected because it restricts the AI to predefined capabilities and prevents true dynamic responsiveness to arbitrary user prompts.

*Note: This architecture is the selected approach for our initial V2 implementation. It will be continually validated through real-world testing and may be superseded if empirical data suggests a better approach.*
